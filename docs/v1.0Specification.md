# Specification: AI Camera Tracker (Self-Contained Local Pipeline)

## Project Overview
A standalone desktop application designed to process undistorted EXR image sequences, generate dynamic exclusion masks, and solve for a 3D camera track using local AI models (SAM 2, CoTracker3, VGGSfM). The final output is a `.nk` (Nuke script) file containing a pre-configured Nuke 16 Classic 3D scene.

## Architecture & Portable Environment Constraints
- **Environment:** Self-contained **Python Windows Embeddable Distribution** (Python 3.10+). The application must NOT rely on the user's system Python installation or modify system environment variables.
- **Directory Structure:** The application folder will contain a `python/` subdirectory holding the portable python executable and all dependencies (PyTorch, PySide6, etc.).
- **Execution:** The application must be launched via a `launch.bat` script that explicitly calls `.\python\python.exe main.py`.
- **GUI Framework:** PySide6.
- **Hardware Target:** NVIDIA RTX A5000 (24GB VRAM) or equivalent.

## Multiprocessing & Memory Management (Critical)
To prevent PyTorch from locking VRAM on a crash, and to ensure the portable environment spawns correctly, the application MUST strictly separate the UI from the AI inference.
- **Process Spawning in Portable Mode:** Before spawning any workers, the main script must explicitly declare `multiprocessing.set_executable(sys.executable)` to guarantee the spawned process uses the portable Python interpreter, not a system default.
- **Process A (UI Main):** Handles the PySide6 UI, Open3D viewport, EXR reading, and `.nk` script generation. It does not import PyTorch.
- **Process B (Worker Sandbox):** Spawned using `multiprocessing.set_start_method('spawn')`. All PyTorch/AI inference runs exclusively inside this process. 
- **Cancel/Crash Protocol:** If the user clicks "Cancel" or Process B throws a fatal exception, Process A must immediately call `process_b.terminate()` to let the OS reclaim 100% of the VRAM.
- **Sequential Internal VRAM Loop:** Process B must never hold more than one model in memory. It must execute a strict Load $\rightarrow$ Infer $\rightarrow$ Destroy loop:
    1. Load SAM 2 $\rightarrow$ generate masks $\rightarrow$ `del sam_model` $\rightarrow$ `gc.collect()` $\rightarrow$ `torch.cuda.empty_cache()`.
    2. Load CoTracker3 $\rightarrow$ generate tracks $\rightarrow$ `del cotracker_model` $\rightarrow$ `gc.collect()` $\rightarrow$ `torch.cuda.empty_cache()`.
    3. Load VGGSfM $\rightarrow$ solve camera $\rightarrow$ return data to Process A.

## UI / UX Layout (Step-by-Step Flow)
*   **Step 1: Setup & Ingest** (EXR sequence, garbage mattes, metadata).
*   **Step 2: Dynamic Masking** (Scrub timeline, click to generate SAM 2 exclusion masks).
*   **Step 3: Solve** (Trigger Process B inference).
*   **Step 4: 3D Review & Orientation** (Open3D viewport, set ground plane via 3-point average, real-world scaling via 2-point distance input).
*   **Step 5: Export** (Generate Nuke 16 `.nk` script with Classic 3D nodes: Read, Camera, PointCloud, Scene, ScanlineRender).

## Phase 1 Executed Architecture Additions
- **Image Ingestion:** Driven strictly by `OpenImageIO` for native EXR handling.
- **Memory Protection:** UI implements strict lazy loading (storing only file paths) combined with a globally shared, 50-frame LRU cache to cap memory growth.
- **Color Space Engine:** Performs localized, mathematically-precise color transformations. Linear EXR data is explicitly converted to standard `uint8` sRGB format via piecewise gamma calculation or ACEScg (AP1) matrixing *before* hitting the UI boundaries.
- **Worker Process Payload:** The multiprocessing sandbox (`Process B`) is strictly passed *absolute file paths* and the *color_space string*, never the raw image array. Process B independently invokes OpenImageIO to parse EXR frames directly from disk, preserving isolated memory spaces.
