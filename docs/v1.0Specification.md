# Specification: AI Camera Tracker (Self-Contained Local Pipeline)

## Project Overview
A standalone desktop application designed to process undistorted EXR image sequences, generate dynamic exclusion masks, and solve for a 3D camera track using local AI models and solvers (SAM 2, CoTracker3, PyCOLMAP). The final output is a `.nk` (Nuke script) file containing a pre-configured Nuke 16 Classic 3D scene.

## Architecture & Portable Environment Constraints
- **Environment:** Self-contained **Python Windows Embeddable Distribution** (Python 3.10+). The application must NOT rely on the user's system Python installation or modify system environment variables.
- **Directory Structure:** The application folder will contain a `python/` subdirectory holding the portable python executable and all dependencies (PyTorch, PySide6, etc.).
- **Execution:** The application must be launched via a `launch.bat` script that explicitly calls `.\python\python.exe main.py`.
- **GUI Framework:** PySide6.
- **Hardware Target:** NVIDIA RTX A5000 (24GB VRAM) or equivalent.

## Multiprocessing & Memory Management (Critical)
To prevent PyTorch from locking VRAM on a crash, and to ensure the portable environment spawns correctly, the application MUST strictly separate the UI from the AI inference.
- **Process Spawning in Portable Mode:** Before spawning any workers, the main script must explicitly declare `multiprocessing.set_executable(sys.executable)` to guarantee the spawned process uses the portable Python interpreter, not a system default.
- **Process A (UI Main):** Handles the PySide6 UI, Open3D viewport, EXR reading, and `.nk` script generation. It does not import PyTorch.
- **Process B (Worker Sandbox):** Spawned using `multiprocessing.set_start_method('spawn')`. All PyTorch/AI inference runs exclusively inside this process. 
- **Cancel/Crash Protocol:** If the user clicks "Cancel" or Process B throws a fatal exception, Process A must immediately call `process_b.terminate()` to let the OS reclaim 100% of the VRAM.
- **Sequential Internal VRAM Loop:** Process B must never hold more than one model in memory. It must execute a strict Load $\rightarrow$ Infer $\rightarrow$ Destroy loop:
    1. Load SAM 2 $\rightarrow$ generate masks $\rightarrow$ `del sam_model` $\rightarrow$ `gc.collect()` $\rightarrow$ `torch.cuda.empty_cache()`.
    2. Load CoTracker3 $\rightarrow$ generate tracks $\rightarrow$ `del cotracker_model` $\rightarrow$ `gc.collect()` $\rightarrow$ `torch.cuda.empty_cache()`.
    3. Load PyCOLMAP $\rightarrow$ solve camera via Triangulation & BA $\rightarrow$ return data to Process A.

## UI / UX Layout (Step-by-Step Flow)
*   **Step 1: Setup & Ingest** (EXR sequence, garbage mattes, metadata).
*   **Step 2: Dynamic Masking** (Scrub timeline, click to generate SAM 2 exclusion masks).
*   **Step 3: Camera Setup & Geometry Preparation** (Parse `cameras.json`, define original plate resolution, set focal length, and automatically calculate Effective Sensor Size for overscanned plates).
*   **Step 4: Solve** (Trigger Process B worker for PyCOLMAP Structure from Motion using the filtered 2D `.npz` tracks and the calculated effective camera sensor parameters).
*   **Step 5: 3D Review & Orientation** (Open3D viewport, set ground plane via 3-point average, real-world scaling via 2-point distance input)[cite: 1].
*   **Step 6: Export** (Generate Nuke 16 `.nk` script with Classic 3D nodes: Read, Camera, PointCloud, Scene, ScanlineRender)[cite: 1].

## Phase 1 Executed Architecture Additions
- **Image Ingestion:** Driven strictly by `OpenImageIO` for native EXR handling.
- **Memory Protection:** UI implements strict lazy loading (storing only file paths) combined with a globally shared, 50-frame LRU cache to cap memory growth.
- **Color Space Engine:** Performs localized, mathematically-precise color transformations. Linear EXR data is explicitly converted to standard `uint8` sRGB format via piecewise gamma calculation or ACEScg (AP1) matrixing *before* hitting the UI boundaries.
- **Worker Process Payload:** The multiprocessing sandbox (`Process B`) is strictly passed *absolute file paths* and the *color_space string*, never the raw image array. Process B independently invokes OpenImageIO to parse EXR frames directly from disk, preserving isolated memory spaces.

## Phase 2 & 2.5 Executed Architecture Additions
- **Explicit Project Management Structure:** The application utilizes an explicit, user-defined project hierarchy instead of implicit folder generation. Creating a new project generates the following structure:
  - `undistorted_source/`: A dedicated directory to copy/move the primary EXR frame sequence.
  - `source_AICameraSolver_Data/`: The isolated working directory for the background daemon to cache generated `proxies/`, binarized `user_masks/`, and SAM 2 `masks/`.
  - `user_masks_source/`: (Created on demand) A directory to copy/move raw custom user masks prior to daemon ingestion.
  - `project.json`: A state-serialization file that actively tracks loaded EXR sequences, active colorspace, proxy resolution choice, extracted native sizes, and other workspace variables to ensure seamless persistence across sessions.
- **Persistent Masking Daemon (Bi-Directional IPC):** Instead of one-shot execution, SAM 2 masking runs via a persistent background daemon communicating with the PySide6 UI over bi-directional queues. This allows for real-time, zero-latency mask previews per-frame when clicking, while still adhering to strict memory isolation.
- **Multi-Object Tracking:** The UI supports adding Positive points (Left Click), Negative points (Right Click), and Deleting points (Middle Click). It also features an Object ID Spinbox, allowing the user to map clicks to distinct discrete objects in SAM 2 logic. SAM 2 generates masks for all distinct objects internally, which are then combined via Boolean Union to create the master exclusion mask.
- **Point Persistence:** Real-time persistence of click data to `clicks.json` within the project directory. The UI dynamically reconstructs the saved clicks and automatically pushes them to the SAM 2 daemon upon masking tab activation, preserving the workspace across sessions.
- **Upfront Multithreaded Proxy Generation:** Proxy JPEG creation and alpha separation are executed immediately during the Import Phase via a PySide6 `QThread` (dropping the GIL). This allows all UI viewers (Setup, Masking, Tracking) to exclusively stream lightweight JPEGs off disk for massive scrubbing performance gains instead of repeatedly parsing floating-point EXRs.
- **Dynamic Proxy Resolution & Coordinate Scaling:** The user can select their desired proxy resolution (1024px, 1536px, 1920px) during import and can seamlessly swap resolutions on the fly. PySide6 viewer clicks are scaled into absolute *native EXR coordinates* prior to saving, and the SAM 2 Daemon accurately downscales those native coordinates back into whatever the active proxy resolution is before tensor translation, ensuring flawlessly consistent object tracking regardless of viewport resolution changes.
- **Memory Optimization (Alpha Channel Extraction):** The proxy generator dumps alpha channels as individual 8-bit PNGs (`alphas/alpha_XXXXX.png`) directly during import instead of bloating system RAM with raw 4K floating-point arrays. The daemon dynamically reads these files on demand during final alpha propagation and mask combination.
- **Keyframe UI Integration:** The PySide6 timeline `QSlider` is subclassed to dynamically paint bright yellow tick marks across the timeline groove wherever click data exists. Quick-jump navigation buttons (`|< Prev Keyframe`, `Next Keyframe >|`) allow instantaneous scrubbing to critical points.
- **Bi-Directional Sequence Propagation:** The SAM 2 Daemon forces propagation passes both forwards and backwards from user-defined keyframes to guarantee that masks accurately extend across the entire EXR sequence regardless of where the keyframes were placed.
- **Dynamic Array Dimension Handling:** SAM 2 mask arrays are instantiated dynamically using runtime tensor output dimensions rather than hardcoded arrays, preventing aspect-ratio mismatch crashes on non-standard video formats.
- **Object ID Colorization:** The PySide6 UI translates unique SAM 2 Object IDs into visually distinct HSV hues (using a 137.508 degree offset) during proxy playback, enabling the user to immediately identify overlapping distinct objects.
- **Custom User Masks (B/W Sequences):** The daemon natively ingests directory-based sequences of user-provided Black/White pre-rendered masks. These are cached locally as binarized 8-bit PNGs, resized via nearest-neighbor to match EXR aspect-ratios dynamically, and compounded directly into the master Boolean Union (displayed as Ruby Red colorization).
- **Download Automation & Setup:** The `DownloadWorker` dynamically creates subdirectories to fetch and correctly place PyTorch weight files, preparing the exact local environment for the Phase 3 models.
- **VRAM Constraint Adherence:** When the user transitions from Masking to the 3D Solver phase, the Persistent Masking Daemon explicitly receives an exit command, executes `del sam_model`, `gc.collect()`, and `torch.cuda.empty_cache()` to completely clear the VRAM before CoTracker3 is loaded.
## Phase 3 Executed Architecture Additions

- **Tracking UI Integration**: Added a dedicated 'Tracking' tab, utilizing a custom `SequenceViewerWidget` to visually overlay the generated 2D tracked points dynamically over the image sequence timeline.
- **Isolated PyTorch Worker**: CoTracker3 inference is driven by an independent multiprocessing worker (`run_tracking_worker`) spawned by the UI to strictly enforce memory boundaries and ensure main thread responsiveness.
- **Adaptive Multi-Pass Bi-Directional Tracking (Continuous Density)**: To prevent tracks from leaving the field of view during extreme camera moves (pans, tilts), the worker utilizes a multi-pass strategy while staying under the strict 500MB VRAM limit. 
  - **Pass 1 & 2 (Forward/Backward)**: 1,024 points are sampled from frame 0 and tracked forward. 1,024 points are sampled from the last frame and tracked backward (reversing the proxy sequence during inference, then flipping the time axis back).
  - **Mid-Sequence Density Respawning**: After passes 1 and 2, the tracker analyzes the active point density across the sequence. If the active track count drops below 35% of the total budget on any frame, it targets the lowest-density frame and respawns 2,048 additional trackers outwards from that point (both forwards and backwards).
  - **Union**: All passes are mathematically merged along the point dimension, yielding a continuous, highly dense point cloud of up to 4,096 tracks regardless of camera movement.
- **Model Selection — `CoTrackerOnlinePredictor` with `scaled_online.pth`**: The offline `CoTrackerPredictor` (which requires the entire video as a single `(B, T, C, H, W)` tensor to be resident in VRAM simultaneously, plus a full-resolution `F.interpolate` copy, making it unsuitable for long sequences) is explicitly **not used**. The application uses `CoTrackerOnlinePredictor` with the `scaled_online.pth` checkpoint, which is architected for streaming inference.
- **Correct Online Streaming Protocol (Overlapping Sliding Window)**: The `CoTrackerOnlinePredictor` requires chunks of exactly `window_len=16` frames, not `step=8`. Each successive chunk slides forward by `step=8` frames, maintaining an **8-frame overlap** with the previous chunk. This overlap is mandatory: the model's internal state machine uses `coords_predicted[:, ind:ind+overlap]` to initialize the next window's positional state. Passing `step`-sized chunks instead of `window_len`-sized chunks results in empty tensor slices and a `RuntimeError`. The correct call sequence is:
  1. Call with `is_first_step=True` to initialize `online_ind=0` and register query points.
  2. Call with `is_first_step=False` for every window (including the first) to run the transformer forward pass.
  3. The model internally increments `online_ind += step` and accumulates predictions across calls.
- **Per-Chunk CPU→GPU Streaming (System RAM Protection)**: Each 16-frame window is read from disk directly into a small `(1, 16, 3, H, W)` float32 CPU tensor, one frame at a time, avoiding a large pre-allocated RAM buffer. The tensor is then cast to `float16` and moved to CUDA immediately before inference. After inference, `del chunk_tensor` releases it immediately. At no point is the full video sequence held in memory.
- **Mixed Precision Inference (`float16` + AMP)**: The entire inference block is wrapped in `torch.inference_mode()` (which completely disables gradient graph tracking, stricter than `torch.no_grad()`) and `torch.autocast(device_type='cuda', dtype=torch.float16)`. The video tensor is explicitly cast to `float16` before being moved to GPU to halve VRAM bandwidth and activation memory. With this configuration, VRAM usage stays flat at approximately **500MB** regardless of sequence length.
- **Mathematical Upscaling**: Because inference runs on the proxy images, the generated 2D coordinates are dynamically upscaled using the calculated `Native_EXR_Dimensions / Proxy_Dimensions` ratio to map the tracks back to native 4K resolution space before filtering.
- **Strict Collision Filtering**: The upscaled track coordinates are iteratively filtered against the native-resolution Master Exclusion Masks. If an upscaled point drifts into a masked exclusion zone during any frame of its trajectory, the entire track history for that point is permanently deleted to guarantee pristine static background data.
- **Strict VRAM Cleanup Protocol**: Upon serialization of the filtered tracks to `tracks.npz`, the worker strictly performs `del cotracker_model`, `gc.collect()`, and `torch.cuda.empty_cache()` to completely clear the VRAM environment prior to advancing to the solver phase.
- **SAM 2 Daemon Lifecycle (Tab-Switch Unload)**: The Persistent Masking Daemon is automatically and immediately shut down whenever the user navigates away from the Masking tab (to any other tab). This ensures zero VRAM contention when transitioning into the Tracking phase.
- **IPC Message Routing**: The `poll_queue` timer on the UI now correctly routes `track_progress`, `track_done`, and `error` status messages from the tracking worker, providing a live progress indicator in the loading overlay and restoring the UI to an interactive state upon completion or failure.

## Phase 4 Pre-Requisites: Camera Logic & Overscan Math
To guarantee the generated 3D camera locks perfectly to the background plate in Nuke, the application must configure precise camera sensor parameters before initiating the PyCOLMAP solver.

- **External Camera Database (`cameras.json`):** The application relies on an external configuration file (`cameras.json`) located in the project or application root. This file catalogs major cinema and prosumer camera formats (Make, Model, Format, Sensor Width in mm, and Sensor Height in mm). Utilizing an external JSON allows users to seamlessly add custom sensor profiles without modifying the core Python architecture.
- **Symmetric Bounding Box Pipeline Rule:** Because the AI solver and Nuke's Camera node both assume the optical center (principal point) of the lens sits at the exact mathematical center of the digital canvas `(Width / 2, Height / 2)`, **the incoming undistorted EXR sequence must have a symmetrically padded bounding box**. If the undistortion pipeline expanded the overscan asymmetrically, the 3D projection will slide.
- **Effective Sensor Size Calculation (Overscan Math):** Physical sensor dimensions cannot be fed directly into the solver if the plate has been undistorted and overscanned. Because the field of view remains constant while the digital canvas expands, the application must calculate the "Effective Sensor Size."
  - **User Inputs Required:** Prior to solving, the user must input the **Original Plate Resolution** (pre-undistortion), select their **Camera Preset** from the JSON, and input the **Focal Length**.
  - **The Calculation:** The application automatically scales the physical sensor width to match the overscan ratio using the following formula: 
    `Effective Sensor Width = Physical Sensor Width * (Undistorted EXR Width / Original Plate Width)`
  - **Solver Ingestion:** This dynamically calculated `Effective Sensor Width` (along with the corresponding Effective Height and Focal Length) is the exact parameter passed into the Phase 4 PyCOLMAP worker process and ultimately written to the Nuke `.nk` Camera node.

### "Best Guess" Fallback Protocols (Missing Metadata)
In production environments, plate metadata is frequently stripped or unavailable by the time it reaches the tracking department. Because 3D tracking mathematically relies on the **Field of View (FOV)**—the ratio between Sensor Size and Focal Length—the application can safely substitute arbitrary values to maintain the correct FOV without breaking the 3D projection. 

To prevent pipeline bottlenecks, the Camera Setup tab includes the following explicit fallback mechanisms:

- **Unknown Camera / Sensor:** The `cameras.json` includes "Generic" profiles (e.g., standard 36mm Full Frame). The user can select this generic profile to lock the math to a standard baseline.
- **Cropped Plate / Unknown Original Resolution:** If the user is tracking stock footage, an editorial punch-in, or a plate that was cropped from its original sensor readout (e.g., Open Gate cropped to UHD) without documentation, they cannot calculate the true sensor crop factor. The UI provides an `[x] Unknown / Assume No Crop` toggle. 
  - **Logic:** When engaged, the application assumes the incoming EXR resolution represents the entire physical sensor area (a 1.0 ratio). The `Effective Sensor Size` is treated as identical to the selected physical sensor size.
- **Unknown Focal Length:** The UI provides an `[x] Auto-Estimate` toggle next to the focal length input.
  - **Logic:** When engaged, the application passes an estimation flag to the PyCOLMAP worker instead of a hardcoded integer. PyCOLMAP will utilize its internal bundle adjustment algorithms to analyze track parallax and dynamically calculate the best mathematical focal length relative to the provided sensor size.

> **Warning to Users (Downstream Trade-offs):** 
> The UI must explicitly inform the user that while these "Best Guess" fallbacks will produce a 100% accurate mathematical projection for the background plate, the physical properties of the generated camera are fabricated. Downstream tasks relying on real-world lens physics (such as physically-based rendering, Z-Defocus, or deep compositing) will behave according to the *guessed* lens metrics, not the physical lens used.

## Phase 4: 3D Solving & Scene Orientation

This phase translates the filtered 2D track data into a 3D point cloud and camera path, and provides the user with the tools necessary to orient that mathematical data into a usable real-world scale before export.

### 4.1: The PyCOLMAP Worker Process (Solver)
This is the final isolated multiprocessing worker, responsible for the heavy Structure from Motion, bundle adjustment and 3D triangulation.
* **Camera Setup State Serialization:** All camera settings (Physical dimensions, overrides, toggles) and UI states are cached directly into `project.json` upon save, allowing instantaneous restoration of the Camera Setup tab across user sessions.
* **Data Ingestion:** Reads the filtered 2D `.npz` tracks from Phase 3, and the calculated `Effective Sensor Size` and `Focal Length` from the Camera Setup tab.
* **Camera Initialization:** Initializes a PyCOLMAP `PINHOLE` camera using the dynamically calculated Effective Sensor parameters and Focal Length to establish the correct Field of View.
* **Track Reliability Filtering:** Before passing data to PyCOLMAP, the worker executes a vectorized reliability algorithm on the CoTracker3 data. It evaluates track lifespan against acceleration jitter (second derivative of motion) to score stability. It then divides the image into a 24x24 spatial grid and strictly selects the top 3 tracks per cell. This ensures the solver receives a highly stable, non-fracturing baseline of ~800 evenly distributed tracks.
* **Fallback Handling:** If the user checked "Auto-Estimate" for the focal length, configures PyCOLMAP's Bundle Adjustment options to optimize and solve for the focal length.
* **Relaxed Initializer Constraints (AI-Track Adaptation):** Standard PyCOLMAP photogrammetry rigidly demands >16-degree triangulation angles for scene initialization, which instantly fails on nodal pans or subtle dolly movements typical in AI video tracking. The worker aggressively forces initialization by relaxing `init_min_tri_angle`, `triangulation.min_angle`, and `filter_min_tri_angle` to 0.1 degrees. This allows near-zero parallax sequences to successfully seed the initial 3D bundle adjustment without PyCOLMAP immediately rejecting the points.
* **Triangulation & BA:** Executes PyCOLMAP's incremental mapping (triangulation and bundle adjustment) purely on the provided 2D tracks by creating an exhaustive two-view match database.
* **Output Serialization:** Extracts the final 3D point cloud, camera rotation matrices, camera translation vectors, the finalized focal length, and the per-track reprojection errors from the PyCOLMAP reconstruction and saves them to `solve_data.npz`.

### 4.2: Open3D Viewport Integration (UI)
Once the worker process completes, the application must visualize the `solve_data.npz` within the main UI.
* **Dual View Architecture:** The Solve tab features a split-screen layout. The right pane embeds an interactive `Open3D` visualizer to render the 3D point cloud and enlarged camera frustums for each frame. The left pane provides a 2D playback sequence overlaying the decimated tracks.
* **Synchronized Interaction:** The 2D viewer and 3D camera path are synchronized in lockstep via a unified timeline scrubber, allowing the user to precisely correlate 2D image features with 3D points.
* **Reprojection Error Filtering (Error Correction):** The UI reads the Reprojection Error metrics generated by PyCOLMAP. A threshold slider is provided to dynamically cull/delete "bad" tracking points that drift too far from their calculated 3D coordinates. This filtering occurs instantaneously in both the 3D point cloud and the 2D viewer (where points dynamically turn Red if they fail the tolerance threshold or Green if they pass).
* **Accurate FOV Visualization (Dynamic Letterboxing):** To perfectly replicate the camera's true Field of View inside the arbitrary, resizable bounds of a PySide6 UI container, the Open3D pinhole camera matrices are dynamically recalculated on every frame. The system reads the active physical render dimensions from Open3D's view controller, calculates the independent aspect-ratio scaling factors against the source plate dimensions (`render_W / plate_W`, `render_H / plate_H`), and applies these scales directly to the mathematical focal length (`fx`, `fy`). The optical center (`cx`, `cy`) is strictly locked to the center of the render dimensions. This entirely prevents Open3D's default auto-stretching behavior and maintains a perfect 1:1 perspective match regardless of how the user reshapes the application window.

### 4.3: Scene Orientation & Scale Toolset (Math)
Because SfM algorithms output data in arbitrary coordinate space and scale, the user must define the real-world alignment before exporting.
* **Ground Plane Definition (Y=0):** A UI mechanic allowing the user to select 3 or more distinct tracks in the Open3D viewport. The application calculates the mathematical average of these points to define the true flat floor, rotating the entire point cloud and camera matrices to align with the global Y-axis.
* **Real-World Scaling:** A tool allowing the user to select 2 distinct points in the viewport and input a known real-world distance between them (e.g., "1.5m"). The application calculates a global scale multiplier and applies it to the entire scene and camera translation matrices to lock the scene to physical measurements.
* **Origin Offset:** A final adjustment allowing the user to select a single point to act as the `(0, 0, 0)` world origin, shifting the entire scene's global coordinates accordingly.

### 4.3.5: AI Depth Verification & Spatial Smoothing
Because SfM triangulation is inherently noisy—particularly on sequences with limited parallax (dolly push-ins, subtle camera moves)—individual 3D points can be triangulated to wildly incorrect depths even when the overall point cloud shape is qualitatively correct. This phase uses a monocular AI depth model as a secondary verification signal to spatially smooth the solved point cloud along each camera's optical rays.

* **Model & Local Inference (Offline Caching):** The system uses **Depth Anything V2 (Small)** via the HuggingFace `transformers` pipeline (`depth-estimation`). All model weights are loaded with `local_files_only=True` for zero network traffic. 
* **Offline Depth Sequence Generation:** To maximize performance during the 3D solving phase, depth inference is executed as an offline preprocessing step in the Setup tab. The `AIDepthSequenceWorker` processes the entire proxy sequence and caches raw floating-point depth maps as `.npy` files into a dedicated `depth_proxies/` directory. This isolates PyTorch VRAM allocation to the Setup phase and makes spatial smoothing instantaneous.
* **Multi-Frame Smoothing Modes:** By leveraging the cached depth sequence, the spatial smoothing operation executes instantly on the main UI thread. The system supports three multi-frame regression modes:
  - **Best View (Fast):** For each 3D point, calculates the single optimal frame where the point is closest to the camera's optical center, and applies the depth correction only from that highly reliable frame.
  - **N-Keyframe Average:** Samples 5 evenly spaced frames, calculates target depth in all visible keyframes, and averages the target 3D coordinate.
  - **Exhaustive Median (Slow):** Calculates the target 3D coordinate across *every single frame* where the point is visible, and takes the robust geometric median for maximum occlusion resistance.
* **Camera-Space Depth Alignment (Robust Linear Regression):** The AI depth map outputs relative disparity values with arbitrary scale and shift. To align these to the SfM solver's metric depth:
  1. Each triangulated 3D point (filtered by `points_mask`) is projected into **camera space** using the first camera's rotation matrix $R$ and translation vector $T$: $P_{cam} = R \cdot P_{world} + T$. The camera-space Z component $P_{cam}[2]$ is the true SfM depth along the optical axis.
  2. The AI disparity at the point's 2D pixel location is converted to relative depth via $z_{ai} = 1 / (d + \epsilon)$.
  3. A **robust least-squares regression** (`scipy.optimize.least_squares` with `loss='soft_l1'`) fits the affine transform $z_{sfm} = m \cdot z_{ai} + c$, with the constraint $m > 0$ to mathematically guarantee the depth map can never be inverted. The Soft-L1 (Huber) loss function automatically downweights massive SfM outliers that would otherwise corrupt a standard linear regression.
* **Optical Ray Scaling (Projection-Preserving Blend):** To adjust a point's depth without altering its 2D screen position:
  1. The point is transformed to camera space: $P_{cam} = R \cdot P_{world} + T$.
  2. The blended target depth is computed: $z_{new} = z_{sfm} \cdot (1 - \alpha) + z_{target} \cdot \alpha$, where $\alpha$ is the user-controlled blend factor.
  3. The **entire camera-space vector** is uniformly scaled by $z_{new} / z_{current}$, preserving the ray direction and therefore the 2D projection.
  4. The result is transformed back to world space: $P_{world} = R^T \cdot (P_{cam,new} - T)$.
  This is critical: naively modifying only the world-Z component would tear apart points that are spatially adjacent in 2D, because the camera's optical axis is almost never aligned with the world Z-axis.
* **UI Controls:**
  - **"Generate Depth Sequence" (Setup Tab):** Caches the floating point depth maps to disk.
  - **"AI Spatial Smoothing" (Solve Tab):** Instantly runs the multi-frame regression and blend pipeline.
  - **Mode Selector:** Dropdown to choose between Best View, N-Keyframe, and Exhaustive Median modes.
  - **"Smoothing Strength" Slider (0–100%):** Controls the blend factor $\alpha$. At 0%, points remain at their original SfM depths. At 100%, points are snapped entirely onto the AI depth surface. Default: 50%.
* **Data Integrity Architecture (Raw vs Master):** The solver strictly delineates between immutable base data and iterative mathematical modifications:
  - `solve_data_raw.npz`: An untouchable, immutable backup containing the exact, raw 3D vectors generated by the VGGSfM solver.
  - `solve_data.npz`: The active master file. AI spatial smoothing directly modifies this file. 
  - When a user applies Orientation Constraints or clicks "Revert to Raw Solve", the mathematical transformation reads the clean points from `solve_data_raw.npz`, transforms them, and safely overwrites the master `solve_data.npz`. This strict pipeline mathematically prevents infinity loop corruption (`NaN` accumulation) resulting from applying smoothing multiple times.
* **UI/UX Solve Tab Restructuring:** Controls are segmented into distinct visual blocks tracking logical order of operations:
  - **1. Base Structure:** Houses the primary VGGSfM solve triggers.
  - **2. AI Spatial Refinement:** Contains Depth Generation, AI smoothing modes, and blend sliders.
  - **3. Data Management:** Houses non-destructive hot-swap previews and the raw solve reversion system.
* **Non-Destructive Viewport Hot-Swapping:** A visual toggle lets the user instantly hot-swap the 3D viewport point-cloud between the Raw state and the Smoothed state. To preserve visual continuity, the Open3D pinhole camera matrices (Extrinsic, Intrinsic, FOV) are cached in memory right before the geometry is flushed, and mathematically re-injected microseconds after the new points are loaded, preventing the view from snapping/resetting.
* **2D Depth Previewer (Thermal Rendering):** The 2D Sequence Viewer features an instant RGB/Depth toggle. When viewing depth, the UI reads the offline cached floating-point `.npy` files, normalizes their global MIN/MAX range mathematically, applies an `inferno` thermal colormap using OpenCV, and converts the arrays into a streaming `QImage` sequence to let the human eye intuitively verify AI perception.
* **Index Mapping (2D↔3D Correlation):** The 3D viewer compacts its internal arrays by stripping un-triangulated points (those where `points_mask` is `False`). A bidirectional lookup table (`track_to_compact` / `compact_to_track`) translates between the 2D viewer's original track indices and the 3D viewer's compacted array indices. This guarantees that clicking a 2D track point highlights the geometrically correct 3D point.
* **Data Loading Order (Race Condition Prevention):** When loading solve data into the 2D viewer, `update_sequence()` is called **before** assigning `track_data` and `track_vis`. This prevents the `SequenceViewerWidget` from auto-loading the raw (pre-decimation) tracking data from `tracks.npz`, which would silently overwrite the solver's decimated track arrays and desynchronize 2D/3D indices.
* **Application Lifecycle (VRAM Cleanup):** The `MainWindow` implements a `closeEvent` handler that explicitly terminates all active daemon processes (`multiprocessing`) and worker threads (`QThread`) before calling `sys.exit(0)`. This prevents orphaned Python processes from holding GPU VRAM after the application window is closed.

### 4.4: AI Reference Geometry Generation (Proxy Geo)
To facilitate accurate downstream compositing (shadow catching, depth occlusion, projection mapping), the application provides an automated workflow to generate low-resolution 3D reference geometry of the scene environment.

* **Monocular Depth Inference:** The system extracts relative depth maps from the 2D proxy sequence using a local daemon process running a lightweight depth model.
* **Metric Depth Alignment (Disparity Space):** The scale-less AI depth maps are metrically anchored to the VGGSfM sparse point cloud. To prevent `1/0` mathematical explosions caused by deep backgrounds, the affine fit is calculated strictly in **disparity space** (`1/Z_sfm = a * disp_ai + b`), converting the AI output into perfectly scaled metric depth maps.
* **Screen-Space Depth Meshing:** The system avoids traditional computationally heavy TSDF/Poisson point cloud fusion. Instead, it applies a bilateral filter to the depth map to smooth noise while preserving object edges. It then treats the 2D pixel grid as a literal mesh, connecting adjacent pixels into quads/triangles to create a 2.5D topological "shell". Triangles that stretch across depth discontinuities (background/foreground separation) are culled via a strict percentage threshold.
* **Volumetric Stacking:** Using the scaled camera focal length and solved transform matrices, each 2.5D shell is back-projected into world space. All frames (subsampled temporally) are stacked together into a single master scene mesh. Vertices closely matching true SfM points are snapped directly to their 3D coordinates.
* **Smoothing & Intelligent Decimation:** The raw stacked geometry is cleaned of degenerate/duplicate geometry and smoothed using **Taubin Smoothing**, which irons out noise spikes without shrinking the volume of the mesh. Finally, the user iteratively decimates the mesh to a target resolution using a **Quadric Error Metric (QEM)** algorithm, which intelligently collapses flat regions into massive triangles while preserving dense detail around structural edges.

## Phase 5 Executed Architecture Additions (Export Subsystems)
The application translates the oriented 3D data into standard compositing and 3D application formats (UI Step 6). Because the application runs in an isolated Python environment without heavy C++ SDKs for binary formats (like Alembic or `.blend`), all exports rely on plaintext script generation.

### 5.1 Nuke Script Generation (`.nk`)
* **Node Assembly:** Generate a standard Nuke 16 plaintext TCL script.
* **Nodes Included:** A `Read` node pointing to the undistorted EXRs, a `Camera` node (with animated translate/rotate matrices, effective sensor size, and focal length), a `PointCloud` node for the 3D tracks, a `Scene` node, and a `ScanlineRender` node.

### 5.2 Blender Macro Generation (`.py`)
* **Execution Method:** The application generates a Python script (`build_vfx_scene.py`). The user executes this script inside Blender's Text Editor to automatically construct the tracking environment.
* **Scene Configuration:** The script utilizes the `bpy` API to configure the global render resolution, frame rate, and timeline range to match the ingested plate.
* **Camera Integration:** The script generates a physical Blender camera, applies the calculated Effective Sensor Dimensions and Focal Length, bakes the 3D translation and rotation matrices into location/rotation keyframes, and attaches the proxy image sequence to the camera's Background Images array for immediate viewport verification.
* **Point Cloud Construction:** To maintain viewport performance and keep the Outliner clean, the script parses the `.npz` point cloud data and generates a single static Mesh object, where each vertex represents a successfully triangulated 3D track.
